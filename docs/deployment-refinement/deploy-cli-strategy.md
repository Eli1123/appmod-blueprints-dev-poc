# Deploy CLI Strategy: Option D — YAML Config + Auto-Detect + Prompt Fallback

## Problem

The current deployment process is ~8 manual steps involving scattered `export` commands, profile configuration, role creation, template generation, and multiple `task` commands. Environment variables get clobbered by tools like `is assume`, there's no validation before deployment starts, and a single missed export silently breaks everything downstream.

## Solution

Three Taskfile phases (`deploy:setup`, `deploy:verify`, `deploy:run`) backed by a single `deploy-config.yaml` file that serves as the source of truth — replacing fragile shell environment variables.

## Config File

### Template: `deploy-config.example.yaml` (committed to repo)

```yaml
# Copy to deploy-config.yaml and fill in values.
# Any missing values will be auto-detected or prompted during setup.

# Required — no auto-detection possible
account_id: ""          # AWS account ID (12 digits)
aws_access_key_id: ""   # Access key for the target account
aws_secret_access_key: "" # Secret key for the target account

# Auto-detected — override if needed
region: ""              # Default: us-west-2 (from AWS_REGION or aws configure)
branch: ""              # Default: current git branch
resource_prefix: ""     # Default: peeks
aws_profile: ""         # Default: taskcat
```

### Actual config: `deploy-config.yaml` (gitignored)

Generated by `deploy:setup`. Contains resolved values for all fields.

### .gitignore addition

```
deploy-config.yaml
```

## Phase 1: `task deploy:setup`

### What it does

1. Checks if `deploy-config.yaml` exists
2. If yes, reads it with `yq` and uses values as overrides
3. For any empty/missing fields, attempts auto-detection:
   - `account_id` → `aws sts get-caller-identity --query Account --output text`
   - `region` → `$AWS_REGION` → `aws configure get region` → default `us-west-2`
   - `branch` → `git branch --show-current`
   - `resource_prefix` → default `peeks`
   - `aws_profile` → `$AWS_PROFILE` → default `taskcat`
4. For fields that can't be auto-detected (credentials, account_id if no active session), prompts interactively
5. Shows summary of all resolved values and asks for confirmation
6. Writes final `deploy-config.yaml`
7. Configures the AWS profile (`aws configure --profile <profile>`)
8. Creates GitLabCIRole if it doesn't exist (with retry for IAM propagation)
9. Writes `PARTICIPANT_ROLE_ARN` into config (derived from account_id)

### Pseudocode

```bash
setup() {
  # Load existing config if present
  if [ -f deploy-config.yaml ]; then
    ACCOUNT_ID=$(yq '.account_id // ""' deploy-config.yaml)
    REGION=$(yq '.region // ""' deploy-config.yaml)
    BRANCH=$(yq '.branch // ""' deploy-config.yaml)
    # ... etc
  fi

  # Auto-detect missing values
  [ -z "$ACCOUNT_ID" ] && ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text 2>/dev/null || echo "")
  [ -z "$REGION" ] && REGION="${AWS_REGION:-us-west-2}"
  [ -z "$BRANCH" ] && BRANCH=$(git branch --show-current)
  [ -z "$RESOURCE_PREFIX" ] && RESOURCE_PREFIX="peeks"
  [ -z "$AWS_PROFILE" ] && AWS_PROFILE="${AWS_PROFILE:-taskcat}"

  # Prompt for anything still missing
  [ -z "$ACCOUNT_ID" ] && read -p "AWS Account ID: " ACCOUNT_ID
  [ -z "$AWS_ACCESS_KEY_ID" ] && read -p "AWS Access Key ID: " AWS_ACCESS_KEY_ID
  [ -z "$AWS_SECRET_ACCESS_KEY" ] && read -sp "AWS Secret Access Key: " AWS_SECRET_ACCESS_KEY

  # Show summary
  echo "Configuration:"
  echo "  account_id:      $ACCOUNT_ID"
  echo "  region:          $REGION (auto-detected)"
  echo "  branch:          $BRANCH (from git)"
  echo "  resource_prefix: $RESOURCE_PREFIX (default)"
  echo "  aws_profile:     $AWS_PROFILE"
  echo ""
  read -p "Confirm? [Y/n]: " confirm
  [ "$confirm" = "n" ] && exit 1

  # Write config
  # ... write deploy-config.yaml with yq or cat heredoc

  # Configure AWS profile
  aws configure set aws_access_key_id "$AWS_ACCESS_KEY_ID" --profile "$AWS_PROFILE"
  aws configure set aws_secret_access_key "$AWS_SECRET_ACCESS_KEY" --profile "$AWS_PROFILE"
  aws configure set region "$REGION" --profile "$AWS_PROFILE"
  aws configure set output json --profile "$AWS_PROFILE"

  # Create GitLabCIRole (idempotent)
  if ! aws iam get-role --role-name GitLabCIRole --profile "$AWS_PROFILE" &>/dev/null; then
    bash scripts/setup-gitlab-aws-role.sh
    # Retry once after 15s for IAM propagation
    sleep 15
    bash scripts/setup-gitlab-aws-role.sh
  fi

  # Derive and store PARTICIPANT_ROLE_ARN
  PARTICIPANT_ROLE_ARN="arn:aws:iam::${ACCOUNT_ID}:role/GitLabCIRole"
  # ... append to config

  echo "Setup complete. Run: task deploy:verify"
}
```

### Idempotency

- Safe to rerun. Reads existing config, only prompts for missing values.
- GitLabCIRole creation checks if role exists first.
- AWS profile configure overwrites cleanly.

## Phase 2: `task deploy:verify`

### What it does

Non-destructive checks that everything is ready before spending 30+ minutes on a deploy.

### Checks (in order)

```
1. deploy-config.yaml exists and is valid YAML
2. All required fields are non-empty
3. AWS profile works:
   - aws sts get-caller-identity --profile <profile> succeeds
   - Account ID matches config
4. GitLabCIRole exists:
   - aws iam get-role --role-name GitLabCIRole --profile <profile>
5. Branch exists on remote:
   - git ls-remote --heads origin <branch> returns a result
6. Branch is checked out locally:
   - git branch --show-current matches config
7. Template has correct branch:
   - yarn generate-cfn-self (or equivalent)
   - grep WORKSHOP_GIT_BRANCH in generated template matches config branch
8. Security Hub is enabled:
   - aws securityhub describe-hub --profile <profile> succeeds
9. CDK is bootstrapped (optional — can be done in deploy phase):
   - Check for CDKToolkit stack in target account
10. No stale environment variables:
    - Warn if AWS_ACCESS_KEY_ID is set in shell (conflicts with profile-based auth)
    - Warn if WORKSHOP_GIT_BRANCH is set in shell and differs from config
```

### Output

```
✅ Config file valid
✅ AWS credentials working (account: 202906064320)
✅ GitLabCIRole exists
✅ Branch 'crossplane-version-upgrade-v2-2026' exists on remote
✅ Branch checked out locally
✅ Template branch matches config
✅ Security Hub enabled
⚠️  CDK not bootstrapped (will bootstrap during deploy)
⚠️  AWS_ACCESS_KEY_ID set in shell — may conflict with profile auth

Ready to deploy. Run: task deploy:run
```

### Fail behavior

Any ❌ check prints what's wrong and how to fix it, then exits non-zero. User fixes the issue and reruns `task deploy:verify`.

## Phase 3: `task deploy:run`

### What it does

1. Sources all values from `deploy-config.yaml` (NOT from shell environment)
2. Exports them for child processes
3. Runs the deployment sequence:
   - `cdk bootstrap aws://<account_id>/<region> --profile <profile>`
   - `task install`
   - `task taskcat-deploy`

### Pseudocode

```bash
deploy() {
  # Refuse to run without verify
  if [ ! -f .deploy-verified ]; then
    echo "Run 'task deploy:verify' first"
    exit 1
  fi

  # Source config — this is the ONLY source of truth
  export AWS_PROFILE=$(yq '.aws_profile' deploy-config.yaml)
  export AWS_REGION=$(yq '.region' deploy-config.yaml)
  export WORKSHOP_GIT_BRANCH=$(yq '.branch' deploy-config.yaml)
  export RESOURCE_PREFIX=$(yq '.resource_prefix' deploy-config.yaml)
  export PARTICIPANT_ROLE_ARN=$(yq '.participant_role_arn' deploy-config.yaml)
  ACCOUNT_ID=$(yq '.account_id' deploy-config.yaml)

  # CDK bootstrap
  cdk bootstrap "aws://${ACCOUNT_ID}/${AWS_REGION}" --profile "$AWS_PROFILE"

  # Generate template with correct branch
  yarn generate-cfn-self
  cp ./assets/peeks-workshop-team-stack-self.json taskcat/templates/team-stack.yaml

  # Deploy
  task install
  task taskcat-deploy
}
```

### Key design decision

The deploy phase explicitly exports from the YAML file, overriding anything in the shell. This is what prevents `is assume` or stale exports from breaking things.

## Taskfile Integration

```yaml
# Add to Taskfile.yml

  deploy:setup:
    desc: 'Configure deployment — prompts for missing values, auto-detects what it can'
    cmds:
      - bash scripts/deploy-cli.sh setup

  deploy:verify:
    desc: 'Validate deployment config — non-destructive checks before deploying'
    cmds:
      - bash scripts/deploy-cli.sh verify

  deploy:run:
    desc: 'Execute deployment — requires verify to pass first'
    cmds:
      - bash scripts/deploy-cli.sh run

  deploy:status:
    desc: 'Check deployment status — CloudFormation stack state and resource health'
    cmds:
      - bash scripts/deploy-cli.sh status

  deploy:clean:
    desc: 'Clean up failed deployment — deletes stack and orphaned resources'
    cmds:
      - bash scripts/deploy-cli.sh clean
```

All logic lives in `scripts/deploy-cli.sh` with subcommands. Taskfile is just the entry point.

## Implementation: `scripts/deploy-cli.sh`

### Structure

```
scripts/
  deploy-cli.sh          # Main entry point, routes to subcommands
  deploy/
    setup.sh             # Phase 1 logic
    verify.sh            # Phase 2 logic  
    run.sh               # Phase 3 logic
    status.sh            # Status checks
    clean.sh             # Cleanup logic
    config.sh            # Shared config loading/writing helpers
    prompts.sh           # Interactive prompt helpers
```

### Config helper (`config.sh`)

```bash
load_config() {
  CONFIG_FILE="deploy-config.yaml"
  if [ ! -f "$CONFIG_FILE" ]; then
    return 1
  fi
  ACCOUNT_ID=$(yq '.account_id // ""' "$CONFIG_FILE")
  REGION=$(yq '.region // ""' "$CONFIG_FILE")
  BRANCH=$(yq '.branch // ""' "$CONFIG_FILE")
  RESOURCE_PREFIX=$(yq '.resource_prefix // ""' "$CONFIG_FILE")
  AWS_PROFILE_NAME=$(yq '.aws_profile // ""' "$CONFIG_FILE")
  PARTICIPANT_ROLE_ARN=$(yq '.participant_role_arn // ""' "$CONFIG_FILE")
}

export_config() {
  load_config || { echo "No config found. Run: task deploy:setup"; exit 1; }
  export AWS_PROFILE="$AWS_PROFILE_NAME"
  export AWS_REGION="$REGION"
  export WORKSHOP_GIT_BRANCH="$BRANCH"
  export RESOURCE_PREFIX="$RESOURCE_PREFIX"
  export PARTICIPANT_ROLE_ARN="$PARTICIPANT_ROLE_ARN"
}

write_config() {
  cat > deploy-config.yaml << EOF
account_id: "${ACCOUNT_ID}"
region: "${REGION}"
branch: "${BRANCH}"
resource_prefix: "${RESOURCE_PREFIX}"
aws_profile: "${AWS_PROFILE_NAME}"
participant_role_arn: "arn:aws:iam::${ACCOUNT_ID}:role/GitLabCIRole"
EOF
}
```

## Bonus: `deploy:status` and `deploy:clean`

### Status

Reads config, checks:
- CloudFormation stack state
- EKS cluster status (all 3)
- ArgoCD capability state
- AMP scraper state

Useful for debugging stuck deployments without digging through AWS console.

### Clean

For when a deployment fails and leaves orphaned resources:
- Deletes CloudFormation stack
- Runs `task taskcat-clean-deployment-force`
- Clears `.deploy-verified` marker

Prompts for confirmation before any destructive action.

## Migration Path

1. Create `deploy-config.example.yaml` and add `deploy-config.yaml` to `.gitignore`
2. Create `scripts/deploy-cli.sh` and `scripts/deploy/` directory
3. Add Taskfile tasks
4. Document in README: "New deployment flow: `task deploy:setup` → `task deploy:verify` → `task deploy:run`"
5. Keep old manual steps documented as fallback until the new flow is proven

## What This Solves

| Problem | How it's solved |
|---------|----------------|
| `is assume` clobbers env vars | Config file is source of truth, not shell env |
| Forgot to export WORKSHOP_GIT_BRANCH | Setup writes it to config, verify checks it |
| Wrong branch in generated template | Verify checks template output matches config |
| GitLabCIRole doesn't exist yet | Setup creates it with retry for IAM propagation |
| Security Hub not enabled | Verify checks it before deploy |
| Stale AWS_ACCESS_KEY_ID in shell | Verify warns about conflicting env vars |
| 8 manual steps to remember | 3 tasks: setup → verify → run |
| No way to check deployment state | `task deploy:status` |
| Failed deploy leaves orphaned resources | `task deploy:clean` |
